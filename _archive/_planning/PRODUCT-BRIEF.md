## The Product

### One-Sentence Description
An AI-powered voice interview that captures a subject matter expert's institutional knowledge before a role transition — and turns it into structured artifacts their successor and team can actually use.

### The Problem
Every organization has people who know things that aren't written down anywhere. When they leave — for promotion, parental leave, transfer, sabbatical, or departure — that knowledge leaves with them. The documentation either never gets written, or it's a hollow checklist that misses the actual institutional knowledge: the workarounds, the judgment calls, the relationships, the things that only make sense if you've been doing the job.

### The Solution
A structured AI-facilitated interview — conversational, not a form. The AI asks targeted questions, adapts follow-ups based on answers, and generates a set of documents that a successor can actually use. A 20-minute conversation produces what would take days to write manually — if it ever got written at all.

### Who Uses This
- **The SME** — the person with the knowledge. They sit down and have a conversation.
- **The Manager or Team Lead** — the person who initiates the knowledge transfer and receives the output.
- **The Successor** — the person who inherits the role and needs the documents.

### Core Output Artifacts
The interview generates a **Knowledge Package** with four documents:

1. **Successor Brief** — High-level orientation for whoever steps into the role. Written for someone smart who doesn't know the specifics yet. Covers: what the person actually owns (not the job title), what will break without intervention, undocumented knowledge, recommended first-week actions.

2. **FAQ** — The questions the successor will almost certainly ask, answered in the SME's own words. Drawn directly from the interview.

3. **Decision Log** — Judgment calls and institutional decisions that lived in the SME's head. Made visible: what was decided, why, and who to ask if unclear.

4. **Runbook** — Step-by-step operational procedures for recurring tasks the SME owns. Written so someone covering can execute without guessing.

## Architecture & Technical Requirements

### How It Works — Two Modes

**Mode 1: Stub Mode (works out of the box, no API keys)**
- AI questions use browser TTS (`window.speechSynthesis`) or pre-scripted text
- SME responses are typed text (with microphone icon indicating future voice support)
- Follow-up questions are pre-mapped branching paths that simulate adaptive conversation
- Output generation uses client-side templates with the captured text
- **This mode must be fully functional and demonstrate the complete experience end-to-end**

**Mode 2: Live Mode (requires user's own API keys)**
- AI questions generated by LLM (configurable — Claude, GPT, Gemini, etc.)
- Voice synthesis via ElevenLabs API (user provides their own key)
- SME can respond via voice (browser speech-to-text) or typed text
- Follow-up questions are genuinely adaptive — the LLM reads prior answers and probes deeper
- Output generation uses the LLM to synthesize interview transcript into the four artifacts
- **API key configuration should be straightforward** — a settings panel or config file where the user plugs in their keys

### Configuration Requirements
The product must support user-provided API keys for:
- **LLM provider** — at minimum: Anthropic (Claude), OpenAI (GPT). The architecture should make adding providers straightforward.
- **Voice synthesis** — ElevenLabs. Optional — the product works without it (falls back to browser TTS).
- **No keys required to run the stub experience.** This is critical. Someone should be able to clone the repo, open it, and go through the full experience without configuring anything.

### Tech Stack Decisions (for your architect to confirm or override)
The reference products use different approaches:
- Candidate Interview Kit: Claude skill + Node.js scripts + HTML Burner App output
- Burner Apps: Python generator + standalone HTML output
- AgentComms: File-based protocol + Node.js + shell scripts

**Recommendation for SME Brain Dump:** This product has a conversational UI as its core — the interview itself is the product, not just the output. This probably means a web application (HTML/CSS/JS minimum, framework optional) rather than a CLI skill. The architect should decide the right approach, but it must result in something a non-technical user can run.

**Deployment target:** Should work when opened locally (file:// or local server) and be deployable to a hosted environment. The landing page deploys via GitHub Pages.
